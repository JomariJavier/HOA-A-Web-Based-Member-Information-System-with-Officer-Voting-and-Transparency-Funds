I. PROJECT OVERVIEW
====================================================================
The HOA system digitizes and manages all core operations of a
Homeowners Association including: member information management,
contribution tracking, anonymous officer voting, transparent funds
management, and public-relations communication.

Key Problems Being Solved:
  1. Paper records are vulnerable to manipulation, loss, and disputes.
  2. "Show of hands" voting exposes members to retaliation from
     candidates — the system must guarantee ANONYMOUS voting.
  3. Funds management is opaque — the system must enforce TRANSPARENCY
     so every member can see how money is spent.

IV. CORE MODULES (SUBSYSTEMS) — DETAILED BREAKDOWN
====================================================================
There are 6 core modules. Each is described below with its purpose,
explicit CAN / CANNOT DO rules per role, and the key database
entities involved. Treat these as non-negotiable specifications.

--------------------------------------------------------------------
MODULE 1: RBAC LOGIN SYSTEM
--------------------------------------------------------------------
Purpose: Authenticate users and enforce role-based access control.
         Prevents unauthorized access and elevation of privilege.
         Each role can ONLY manage accounts at its own level or below.

  MEMBER — CAN:
    ✓ Log in and log out of their own account
    ✓ Edit their OWN username
    ✓ Change their OWN password
  MEMBER — CANNOT:
    ✗ View, create, edit, or delete ANY other user's account
    ✗ See a list of other users
    ✗ Access any admin or superadmin panels

  ADMIN (Officer) — CAN:
    ✓ Everything a Member can (for their own account)
    ✓ Create new Member accounts
    ✓ Edit existing Member accounts (username, password reset, status)
    ✓ Delete / deactivate Member accounts
    ✓ View a list of all Member accounts
  ADMIN — CANNOT:
    ✗ Create, edit, or delete other Admin accounts
    ✗ Create, edit, or delete Superadmin accounts
    ✗ Elevate a Member to Admin (only Superadmin can)
    ✗ Access Superadmin system configuration

  SUPERADMIN — CAN:
    ✓ Everything for their own account
    ✓ Create new Admin accounts
    ✓ Edit existing Admin accounts
    ✓ Delete / deactivate Admin accounts
    ✓ System-wide configuration and role management
  SUPERADMIN — CANNOT:
    ✗ Directly create, edit, or delete Member accounts
      (that is the Admin's responsibility — separation of duties)

  Security Requirements:
    - Passwords MUST be hashed (bcrypt or equivalent — NEVER plaintext)
    - Session management with secure, HttpOnly tokens
    - Spring Security with role-based endpoint protection
    - Every request must validate the user's role on the backend
    - Failed login attempts should be rate-limited

  Key Entities: User (id, username, password_hash, role, status,
                created_at, updated_at)

--------------------------------------------------------------------
MODULE 2: PERSON INFORMATION SYSTEM (Member Records)
--------------------------------------------------------------------
Purpose: Maintain accurate, secure, and centralized records of all
         HOA members — replacing the old paper-based system.
         This module is ADMIN-ONLY for data management. Members have
         NO access to this module's interface.

  MEMBER — CAN:
    (nothing — Members do NOT have access to this module)
  MEMBER — CANNOT:
    ✗ View member records (not even their own through THIS module)
    ✗ Add, edit, or delete any member information
    ✗ Access the Person Information System pages or API endpoints
    Note: Members manage their own ACCOUNT info via Module 1 (Login),
    NOT through this module. This module is strictly for Admin use.

  ADMIN — CAN:
    ✓ View ALL member records (full directory)
    ✓ Add new member records (name, address, contact, etc.)
    ✓ Edit / update existing member records
    ✓ Delete / deactivate member records
    ✓ Search and filter members by name, status, etc.

  SUPERADMIN — CANNOT:
    ✗ Access this module (member record management is Admin's domain)

  Security Requirements:
    - All changes to member records MUST be logged (audit trail)
    - API endpoints must reject requests from non-Admin roles with 403

  Key Entities: Member (id, full_name, address, contact_number,
                email, join_date, status, contribution_status)

--------------------------------------------------------------------
MODULE 3: VOTING SYSTEM
--------------------------------------------------------------------
Purpose: Enable secure, anonymous elections for HOA officer positions.
         Replaces the "show of hands" method to prevent retaliation.

  MEMBER — CAN:
    ✓ View active and upcoming elections
    ✓ View candidate profiles and information
    ✓ Cast ONE anonymous vote per election (during the voting period)
    ✓ View vote RESULTS after the election closes (aggregated totals)
  MEMBER — CANNOT:
    ✗ Vote more than once per election
    ✗ Change or retract a vote after submission
    ✗ See WHO voted for whom (including their own vote in the results)
    ✗ Manage candidates or election settings

  ADMIN — CAN:
    ✓ Create and configure elections (positions, dates, descriptions)
    ✓ Add and manage candidate information (profiles, positions)
    ✓ Open and close voting periods
    ✓ View aggregated vote results (totals per candidate)
    ✓ Declare and publish final results
  ADMIN — CANNOT:
    ✗ Cast a vote (Admins/Officers are managers, not voters)
    ✗ Modify, tamper with, or delete vote results — EVER
    ✗ View individual voting records (who voted for whom)
    ✗ Trace any vote back to a specific member

  CRITICAL CONSTRAINT — ANONYMITY:
    >>> Voting identity MUST be decoupled from vote records.     <<<
    >>> The system must make it IMPOSSIBLE to trace a vote back  <<<
    >>> to the member who cast it. This is a CORE requirement.   <<<
    >>> NO role — not even Superadmin — can see who voted whom.  <<<

  CRITICAL CONSTRAINT — RESULT INTEGRITY:
    >>> Vote results are READ-ONLY for ALL roles.                <<<
    >>> No user at any level can modify tallies once cast.        <<<
    >>> The database must enforce this via immutable records.     <<<

  Security Requirements:
    - One vote per member per election (enforced at DB level via
      unique constraint on VoteTracker)
    - Vote records must NOT contain the voter's identity
    - Use a separate "has_voted" flag decoupled from the vote record
    - All voting endpoints require authentication
    - Voting period must be enforced server-side (not just UI)

  Key Entities:
    - Election (id, title, description, start_date, end_date, status)
    - Candidate (id, election_id, member_id, position, bio)
    - Vote (id, election_id, candidate_id) ← NO voter_id!
    - VoteTracker (id, election_id, member_id, has_voted) ← separate

MODULE 4: AUDITING SYSTEM (Funds & Budget Management)
--------------------------------------------------------------------
Purpose: Track ALL financial transactions — income (contributions)
         and expenses (projects, maintenance) — with full transparency
         so members can see exactly how funds are being used.

  MEMBER — CAN:
    ✓ View all projects and their details
    ✓ View all recorded expenses and their breakdowns
    ✓ View financial summaries (total income, total expenses, balance)
    ✓ View contribution records
  MEMBER — CANNOT:
    ✗ Add, edit, or delete any financial record
    ✗ Add, edit, or delete any project
    ✗ Access any write/modify endpoints for this module

  ADMIN — CAN:
    ✓ Everything a Member can (view all financial data)
    ✓ Add new project records
    ✓ Edit existing project details
    ✓ Record new expenses with descriptions and categories
    ✓ Edit expense records (corrections are logged)
    ✓ Record incoming contributions / payments
    ✓ Generate financial reports
  ADMIN — CANNOT:
    ✗ Permanently delete any financial record (append-only / soft-delete)
    ✗ Hide transactions from Member view (transparency is mandatory)

  CRITICAL CONSTRAINT — TRANSPARENCY:
    >>> Every financial transaction MUST be visible to ALL Members. <<<
    >>> Officers CANNOT hide, permanently delete, or suppress     <<<
    >>> any financial entries. All edits leave an audit trail.     <<<

  Security Requirements:
    - Financial records should be append-only (soft-delete only)
    - Every transaction logs: who entered it, when, and what changed
    - Data integrity checks in PostgreSQL (constraints, foreign keys)
    - Edit history must be preserved (no silent overwrites)

  Key Entities:
    - AuditRecord (id, type [income/expense], amount, description,
      category, project_id, recorded_by, created_at, updated_at)
    - Project (id, name, description, budget, start_date, status)

--------------------------------------------------------------------
MODULE 5: PUBLIC RELATIONS (PR) SYSTEM
--------------------------------------------------------------------
Purpose: Facilitate communication between officers and members
         through a complaint/feedback system.

  MEMBER — CAN:
    ✓ Post / submit a complaint or feedback
    ✓ View the status and responses to their OWN complaints
  MEMBER — CANNOT:
    ✗ View other members' complaints
    ✗ Respond to or resolve complaints
    ✗ Edit or delete a complaint after submission
    ✗ Post announcements

  ADMIN — CAN:
    ✓ View ALL submitted complaints from all members
    ✓ Respond to and provide feedback on complaints
    ✓ Update the status of complaints (open → in progress → resolved)
    ✓ Create, edit, and publish announcements
    ✓ Pin or archive announcements
  ADMIN — CANNOT:
    ✗ Delete member complaints (complaints are permanent records)
    ✗ Edit the content of a member's original complaint

  Key Entities:
    - Announcement (id, title, content, author_id, published_at,
      is_pinned, is_archived)
    - Complaint (id, member_id, subject, body, status, response,
      created_at, responded_at, responded_by)

  >>> Both Members and Admins have access to the dashboard,        <<<

VI. SECURITY FRAMEWORK (STRIDE MODEL)
====================================================================
Every feature you build must account for these threat categories:

  Spoofing       → Enforce authentication; hash passwords; manage sessions
  Tampering      → Validate ALL inputs; use HTTPS; DB constraints; audit logs
  Repudiation    → Log every action (who, what, when) with timestamps
  Info Disclosure→ RBAC enforcement; encrypt sensitive data; decouple vote identity
  Denial of Svc  → Rate limiting; error handling; connection pooling
  Elev. of Priv  → Backend role validation on EVERY request; never trust the frontend
