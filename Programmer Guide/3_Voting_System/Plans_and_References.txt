====================================================================
VOTING SUBSYSTEM (HOA VOTING ROOM) — Plans & References
====================================================================

PURPOSE
--------------------------------------------------------------------
The Voting Subsystem, or "HOA Voting Room," provides a digital platform
for conducting secure, transparent, and democratic elections within 
the Homeowners Association. It replaces paper ballots with a unified
Material Design 3 interface.

SYSTEM FLOW
--------------------------------------------------------------------
The user navigates through three primary views:

    [ VIEW 1 ] → [ VIEW 2 ]
    Voting Room  Election View
    (List)       (Ballot)
        │            │
        └─────┬──────┘
              ▼
          [ VIEW 3 ]
          Create Poll
          (Admin Only)


====================================================================
VIEW 1: VOTING ROOM (THE DIRECTORY)
====================================================================

WHAT IT DOES:
- Lists all votable elections as discrete Elevated Cards (M3).
- Uses LinearProgressIndicator for voting metrics:
  $Progress = Voters_{Current} / Voters_{Total}$
- Links each item to View 2 (Election View).
- Provides an Extended FAB for View 3 (Create Poll).

DESIGN GUIDELINES:
- Base elevation: Level 1 (Level 2 on hover).
- Active Elections: Use Primary color tokens.
- Concluded Elections: Use Outline tokens (archival de-emphasis).
- FAB: Bottom-right, 16dp padding, "plus" icon + "Create Poll" text.


====================================================================
VIEW 2: ELECTION VIEW (THE BALLOT)
====================================================================

WHAT IT DOES:
- Shows the candidates for a specific election position.
- Uses Identity Cards for nominees with 32dp circular avatars.
- Selection via Radio-List pattern (Outlined Card state).

DESIGN GUIDELINES:
- Typography: Headline Large for names, Body Medium for credentials.
- Constraint: Credentials truncated at 150 characters.
- Selection State: Container transitions to Tonal Palette Primary.
- "Hidden Vote" Protocol: No real-time quantitative metrics shown.
- "Create Poll" button: Also visible here (for Admin) to navigate to View 1.3.
- Finalization: Singular "Confirm Vote" FAB in bottom-right.
  (Logical buffer to mitigate Action Bias).


====================================================================
VIEW 3: CREATE POLL (ADMIN WORKSPACE)
====================================================================

WHAT IT DOES:
- Accessible ONLY to Admin role.
- Collects: Election Title, Start/End Time, and Nominees.

DESIGN GUIDELINES:
- Workspace: Surface Container High.
- Inputs: M3 Outlined Text Fields with Label/Supporting text.
- Chronology: MUST use M3 Date Picker component.
- Nominees: Dynamic List organism with "remove" affordance.
- Constraint: Hard limit of $N \le 5$ nominees.
- Terminal Actions: Button Row (Bottom-right).
  - "Back" (Outlined Button)
  - "Create" (Filled Button)

SYSTEMIC ANALOGY:
The Admin Form acts as a Relational Schema Validator. Just as a 
database rejects a null value in a primary key, the UI should 
reject a poll that lacks a nominee or a valid date range 
before the "Create" action is permitted.


====================================================================
DATA MODEL (Proposed)
====================================================================

Poll/Election:
- id (Long)
- title (String / Position)
- startDate (LocalDateTime)
- endDate (LocalDateTime)
- status (Active, Concluded)
- nominees (List<Nominee>)

Nominee:
- id (Long)
- name (String)
- credentials (String / Bio)
- voteCount (Long - Encrypted/Hidden in UI)

====================================================================
DESIGN CONSISTENCY STANDARDS (M3)
====================================================================
As of the global design sync pass, the Voting Subsystem must adhere to:

1. Universal Header:
   - Must use the `.pis-header` flex-between layout.
   - Includes the "Admin Sandbox" toggle to prototype Officer permissions.
   - Descriptive paragraph text must explain the module's transparency purpose.

2. Content Constraints:
   - All main sections (List cards, Ballot, Admin Form) must be wrapped in 
     a container set to `max-width: 1100px` for optimal readability.

3. Information Content Strategy:
   - Election cards cannot be "barebones."
   - Must show qualitative summaries (Start/End dates in readable format).
   - Must show quantitative metrics (Voter turnout as percentage).
   - Must indicate the number of candidates participating.

4. Form Segmenting:
   - Admin forms must not be a single monolithic list.
   - Divide into logical sections (e.g., "1. Basic Info", "2. Nominees") 
     using M3 Elevated Cards or distinct surface dividers.
